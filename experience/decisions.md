# 架构决策记录 (ADR)

> 记录重要的技术选型决策：选了什么、不选什么、为什么。

## 格式

```
### ADR-001：决策标题

**日期**：YYYY-MM-DD
**状态**：提议 / 已采纳 / 已废弃
**背景**：为什么需要做这个决定
**决策**：选了什么方案
**备选方案**：考虑过但没选的方案
**后果**：这个决定带来了什么影响（正面 + 负面）
```

---

## 记录

### ADR-001：Spring Boot 2.7 而非 3.x

**日期**：2025
**状态**：已采纳
**背景**：毕业设计项目，需要稳定生态和丰富教程
**决策**：选择 Spring Boot 2.7.18（2.x 最后一个稳定版）
**备选方案**：Spring Boot 3.x（需 Java 17+、jakarta 迁移，教程较少）
**后果**：生态成熟，问题好搜；后续升级到 3.x 有一定迁移成本

### ADR-002：MyBatis-Plus 而非 JPA

**日期**：2025
**状态**：已采纳
**背景**：需要灵活 SQL 控制，复杂查询场景多
**决策**：MyBatis-Plus 3.5.5
**备选方案**：Spring Data JPA（自动生成简单 CRUD 方便，但复杂查询力不从心）
**后果**：SQL 完全可控，复杂联表查询用 `@Select` 或 XML 写原生 SQL

### ADR-003：宝塔面板管理服务器

**日期**：2025
**状态**：已采纳
**背景**：运维经验处于实践阶段，需要可视化面板降低操作门槛
**决策**：宝塔面板部署 Spring Boot 应用 + MySQL 管理
**备选方案**：纯命令行 Docker（学习曲线陡峭）、手动安装（配置繁琐）
**后果**：部署效率高，可视化操作安心；依赖面板环境，SSH 受限时只能走终端

### ADR-004：Spring Boot 3.2.5 升级

**日期**：2026-06
**状态**：已采纳
**背景**：知识库项目从零新建，不再受 2.7 生态约束，且 3.x 已成为主流
**决策**：Spring Boot 3.2.5 + Java 17 + Jakarta EE
**备选方案**：继续用 2.7（与 ADR-001 一致但已过时）、Spring Boot 3.0（早期版本不够稳定）
**后果**：Jakarta 迁移一次到位，Security 6.x 用 lambda DSL 更简洁；旧项目 2.7 代码需手动适配

### ADR-005：DeepSeek API 作为 AI 引擎

**日期**：2026-06
**状态**：已采纳
**背景**：知识库需要 AI 对话和内容生成能力，预算敏感
**决策**：DeepSeek API（deepseek-chat）
**备选方案**：OpenAI GPT-4（贵 10-20 倍、需翻墙）、国内其他大模型（智谱/通义千问，API 稳定性差异大）
**后果**：成本极低，中文质量好；无 embeddings 端点，无法做向量检索

### ADR-006：MySQL LIKE 替代向量检索做文档搜索

**日期**：2026-06
**状态**：已采纳
**背景**：DeepSeek 无 embeddings API，需另选文档检索方案
**决策**：AI 提取查询关键词 → MySQL LIKE 多词 OR 匹配
**备选方案**：接入 OpenAI embeddings（增加成本和网络依赖）、Elasticsearch（运维复杂度高）、本地向量模型（效果不确定）
**后果**：零额外成本、零运维；中文匹配精度不如向量检索，需 AI 辅助关键词提取补偿
