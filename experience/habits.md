# 开发习惯

> 个人操作模式和偏好，帮助 AI 理解工作方式。重复 ≥3 次的模式记录于此。

## 格式

```
### 习惯名称

**场景**：什么时候触发
**操作**：具体怎么做
**原因**：为什么这么做
```

---

## 环境操作

### 服务器管理走宝塔终端

**场景**：需要在服务器上执行命令
**操作**：登录宝塔面板 → 终端 → 执行命令
**原因**：SSH 直连受限，宝塔终端稳定可用

### 配置外挂目录

**场景**：Spring Boot 部署时修改配置
**操作**：jar 文件同级创建 `config/` 目录，放入 `application.yml`
**原因**：Spring Boot 外挂配置优先级高于 jar 内配置，改配置不用重新打包

### 数据库备份走宝塔计划任务

**场景**：生产数据库需要定期备份
**操作**：宝塔 → 计划任务 → mysqldump → 每天凌晨执行
**原因**：稳定、可视化管理、有执行日志

## 开发节奏

### 本地验证再部署

**场景**：写完代码准备上线
**操作**：`mvn clean package` 本地确认编译通过 → 上传 jar → 重启服务
**原因**：避免把编译错误的代码传到服务器

### 后端先行开发

**场景**：开始一个新功能模块
**操作**：先定义实体类 + 建表 SQL → 写 Mapper/Service/Controller → 再写前端
**原因**：接口先定好，前后端对接不会返工

### 不改太多文件时不 stash

**场景**：需要临时切分支，但有未提交的改动
**操作**：如果改动文件 ≤3 个，直接 `git add` + `git commit -m "WIP"`；多了才 `git stash`
**原因**：stash 恢复时容易冲突，小改动直接提交更省事

## Git 习惯

### 精确 add，不用 git add .

**场景**：准备提交代码
**操作**：`git status` 看清单 → `git add src/main/java/.../Xxx.java` 逐个加
**原因**：避免把临时文件、本地配置、不需要的改动混入提交

### push 前确认

**场景**：代码已 commit，准备推送
**操作**：再跑一次 `git status` + `git log --oneline -3` 确认改动范围
**原因**：远程仓库一旦推了就不能改，多看一眼少出错

### commit message 用中文

**场景**：写 commit message
**操作**：用中文简述改动
**原因**：自己看得更直观，毕业设计不需要英文规范

## 文档习惯

### ER 图先出再写 SQL

**场景**：设计新表结构
**操作**：先画 ER 图理清实体关系 → 确认无误 → 再写 CREATE TABLE
**原因**：在图上改关系比改 SQL 快，避免建完表发现关联不对

### 改动后总结存档

**场景**：一次开发告一段落
**操作**：让 AI 输出分条总结（改动文件 + 改动原因）
**原因**：方便回顾，写毕设论文时能追溯做了什么
